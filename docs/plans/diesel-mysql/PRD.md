# MySQL/MariaDB Backend for yauth

## Overview

Add a `DieselMysqlBackend` to yauth, enabling MySQL and MariaDB as first-class database backends alongside PostgreSQL, SQLite/Turso, and in-memory. The backend architecture, repository traits, domain types, and MySQL DDL generation are already complete — this project implements the concrete backend module and wires it into the build.

## Core Features

1. **DieselMysqlBackend struct** — Connection pool creation via `diesel-async` with `AsyncMysqlConnection` (deadpool), configurable pool size, `new(url)` constructor matching the existing backend API.

2. **Schema and migrations** — MySQL-flavored DDL (already generated by `schema/mysql.rs`) applied via `migrate()`. Feature-gated migration strings matching the plugin flag structure used by diesel_pg.

3. **Full repository trait implementation** — All sealed repository traits implemented against MySQL, following the same patterns as `diesel_pg`:
   - Core: UserRepository, SessionRepository, AuditLogRepository, ChallengeRepository, RateLimitRepository, RevocationRepository, SessionOpsRepository
   - Feature-gated: PasswordRepository, PasskeyRepository, TotpRepository, BackupCodeRepository, OauthAccountRepository, OauthStateRepository, ApiKeyRepository, RefreshTokenRepository, MagicLinkRepository, AccountLockRepository, UnlockTokenRepository, WebhookRepository, WebhookDeliveryRepository, and OAuth2 server repos

4. **MySQL-specific type handling** — UUID stored as `CHAR(36)` (string conversion in models), booleans as `TINYINT(1)`, `DATETIME` without timezone, `JSON` (not JSONB). UUIDs generated in Rust (`Uuid::now_v7()`) since MySQL has no native UUID type.

5. **Diesel common integration** — Extend `diesel_common` `PoolAccess` trait to support the MySQL async pool, reuse `diesel_err()` and `diesel_conflict()` helpers (with a MySQL-specific conflict mapper similar to `diesel_conflict_sqlite()`).

6. **Redis caching compatibility** — The Redis decorator wraps repository traits, not backends. MySQL backend should work with `with_redis()` out of the box — no extra work needed, but verify it.

7. **Integration tests** — Pentest/integration test suite extended to run against MySQL (Docker Compose service). Memory + diesel_pg + diesel_mysql all exercised.

8. **Repository conformance test suite** — A new, dedicated test module that exercises every repository trait method against every backend and asserts identical behavior. The existing pentest suite covers OWASP security scenarios via HTTP, but there are no tests that directly call repository trait methods and verify results are consistent across backends. This suite fills that gap:
   - **Scope:** Every method on every repository trait (core + feature-gated). Each test creates data, reads it back, updates, deletes, and verifies error cases (NotFound, Conflict).
   - **Parameterized over backends:** A single set of test functions runs against memory, diesel_pg, diesel_libsql, and diesel_mysql. Uses a shared `test_backends()` helper that returns `Vec<(name, Repositories)>` — skipping unavailable backends (no database URL → skip).
   - **Behavioral assertions:** Tests verify not just "it didn't error" but specific return values, ordering, pagination, expiry behavior, and error variants. If a backend returns results in a different order or silently drops a field, the test catches it.
   - **MySQL-specific edge cases:** UUID round-tripping (CHAR(36) → Uuid), boolean storage (TINYINT(1)), datetime precision (fractional seconds), JSON field read-back, and case-insensitive email uniqueness.

## Technical Decisions

- **diesel-async with deadpool** — same async pool strategy as diesel_pg. The `diesel-async` crate supports MySQL via the `mysql` feature.
- **No RETURNING clause** — MySQL doesn't support `RETURNING`. Use `INSERT` then `SELECT` (or `get_result` via diesel's MySQL support where available). This is the biggest behavioral difference from the PG backend.
- **UUID as CHAR(36)** — stored as strings. The Diesel model layer converts `String ↔ Uuid`. This matches the pattern already established in the schema DDL generator.
- **Configurable database prefix** — `yauth_` table prefix is already handled by the schema system. No MySQL-specific work needed.
- **MariaDB compatibility** — target MySQL 8.0+ and MariaDB 10.6+. Both support `JSON`, `DATETIME(6)` fractional seconds, and InnoDB as default engine. Avoid MySQL-only syntax that breaks MariaDB (e.g., `CHECK` constraints behave differently — use application-level validation).
- **Case-insensitive email** — MySQL's default `utf8mb4_0900_ai_ci` collation is case-insensitive for `VARCHAR`. Email uniqueness works by default. Add a note in docs that collation matters.

## Non-Goals

- **MySQL-specific optimizations** — no MySQL-only features (partitioning, specific indexes) in v1. Keep behavior identical to other backends.
- **Configurable MySQL schema** — PostgreSQL supports `SET search_path` for schema isolation. MySQL doesn't have an equivalent. Skip for now.
- **Connection SSL/TLS configuration** — defer to the connection URL and driver defaults. Don't add MySQL-specific TLS builder options in v1.
- **Read replicas / connection splitting** — out of scope. Single read-write pool only.
