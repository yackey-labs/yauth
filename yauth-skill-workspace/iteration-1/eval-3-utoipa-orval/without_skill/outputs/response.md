# Integrating yauth Into an Existing utoipa + orval Pipeline

## Overview

You have two separate OpenAPI specs: one for your app's API and one for yauth's auth endpoints. The goal is to keep them independent (so yauth upgrades don't break your app's client generation) while ensuring all API calls -- both your app's and yauth's -- carry the session cookie automatically.

---

## 1. Rust Side: Nest yauth's Router Under a Prefix

yauth gives you an `axum::Router<YAuthState>` via `yauth.router()`. Nest it under `/api/auth` (or whatever prefix you prefer), separate from your app's routes.

```rust
use yauth::prelude::*;
use yauth::{YAuthBuilder, create_pool};

#[tokio::main]
async fn main() {
    let database_url = std::env::var("DATABASE_URL").expect("DATABASE_URL required");

    let yauth_config = YAuthConfig {
        base_url: "http://localhost:3000".into(),
        session_cookie_name: "session".into(),
        session_ttl: std::time::Duration::from_secs(86400),
        secure_cookies: false, // true in production
        trusted_origins: vec!["http://localhost:5173".into()],
        ..Default::default()
    };

    let pool = create_pool(&database_url, &yauth_config)
        .expect("Failed to create pool");

    let yauth = YAuthBuilder::new(pool.clone(), yauth_config)
        .with_email_password(Default::default())
        // .with_mfa(Default::default())
        // .with_passkey(passkey_config)
        .build();

    let yauth_router = yauth.router();
    let yauth_state = yauth.into_state();

    // Your existing app router (uses your own AppState)
    let app_router = your_app_routes();

    let app = axum::Router::new()
        // Your app's routes
        .merge(app_router)
        // yauth's auth routes nested under /api/auth
        .nest("/api/auth", yauth_router.with_state(yauth_state.clone()))
        // Make YAuthState available for your app's auth middleware
        .layer(axum::Extension(yauth_state));

    // ... add CORS, start server, etc.
}
```

**Key point:** `yauth_router.with_state(yauth_state)` converts the `Router<YAuthState>` into a `Router<()>`, so it composes cleanly with your existing app router regardless of what state type your app uses.

### Protecting Your App's Routes with yauth's Auth Middleware

To protect your own routes with yauth's session/bearer/api-key authentication, use yauth's `auth_middleware` on your app's routes:

```rust
use axum::middleware::from_fn_with_state;
use yauth::middleware::{auth_middleware, AuthUser};

let protected_app_routes = axum::Router::new()
    .route("/api/items", get(list_items).post(create_item))
    .route("/api/items/:id", get(get_item).put(update_item).delete(delete_item))
    .layer(from_fn_with_state(yauth_state.clone(), auth_middleware));
```

Inside your handlers, extract the authenticated user:

```rust
async fn list_items(
    Extension(user): Extension<AuthUser>,
    // ... your other extractors
) -> impl IntoResponse {
    // user.id, user.email, user.role are available
}
```

## 2. Keep Your OpenAPI Spec Separate from yauth's

**Do not merge yauth's `openapi.json` into your app's spec.** They are separate concerns:

- **Your app's spec** (`openapi.json`): Generated by your existing utoipa setup, covers `/api/items`, `/api/orders`, etc.
- **yauth's client** (`@yackey-labs/yauth-client`): Pre-built npm package with its own generated client that handles all `/api/auth/*` endpoints.

This means:

1. Your existing utoipa + orval pipeline stays exactly as-is. No changes needed.
2. You install `@yackey-labs/yauth-client` as a separate npm dependency for auth calls.

```bash
bun add @yackey-labs/yauth-client
```

## 3. TypeScript: Configure Both Clients to Carry the Session Cookie

### 3a. Configure yauth's Client

yauth's client uses a `customFetch` mutator that defaults to `credentials: "include"` (sending cookies automatically). You just need to call `configureClient` once at app startup:

```typescript
import { configureClient, createYAuthClient } from "@yackey-labs/yauth-client";

// Initialize the yauth client -- baseUrl must match the nest prefix
const authClient = createYAuthClient({
  baseUrl: "/api/auth",
  // credentials defaults to "include", so session cookies are sent automatically
});

// Use it:
const session = await authClient.getSession();
await authClient.emailPassword.login({ email: "user@example.com", password: "..." });
await authClient.logout();
```

The `credentials: "include"` default in yauth's `customFetch` mutator ensures the session cookie is sent with every auth request. No extra configuration needed.

### 3b. Configure Your App's orval Mutator to Also Send Cookies

Your existing orval-generated client needs the same `credentials: "include"` behavior. If you already have a custom mutator, add `credentials: "include"` to it. If not, create one:

```typescript
// src/api/mutator.ts
export const customFetch = async <T>(
  input: RequestInfo,
  init?: RequestInit,
): Promise<T> => {
  const response = await fetch(input, {
    ...init,
    credentials: "include", // This is the critical line -- sends the session cookie
  });

  if (!response.ok) {
    const text = await response.text();
    throw new Error(text || `Request failed with status ${response.status}`);
  }

  const text = await response.text();
  return (text ? JSON.parse(text) : undefined) as T;
};
```

Then reference it in your `orval.config.ts`:

```typescript
import { defineConfig } from "orval";

export default defineConfig({
  myApp: {
    input: {
      target: "./openapi.json",  // Your app's spec, NOT yauth's
    },
    output: {
      mode: "single",
      target: "./src/api/generated.ts",
      client: "fetch",
      baseUrl: false,
      override: {
        mutator: {
          path: "./src/api/mutator.ts",
          name: "customFetch",
        },
      },
    },
  },
});
```

## 4. CORS Configuration

For the session cookie to actually be sent cross-origin (e.g., Vite dev server on `:5173` talking to Axum on `:3000`), configure CORS on the Axum side:

```rust
use tower_http::cors::{CorsLayer, AllowOrigin, AllowHeaders, AllowMethods};
use axum::http::{HeaderName, Method};

let cors = CorsLayer::new()
    .allow_origin(AllowOrigin::list([
        "http://localhost:5173".parse().unwrap(),
    ]))
    .allow_methods([Method::GET, Method::POST, Method::PUT, Method::PATCH, Method::DELETE])
    .allow_headers([
        HeaderName::from_static("content-type"),
        HeaderName::from_static("traceparent"),   // for OTel
        HeaderName::from_static("tracestate"),    // for OTel
        HeaderName::from_static("x-api-key"),     // if using API key auth
    ])
    .allow_credentials(true);  // Required for cookies

let app = axum::Router::new()
    .merge(app_router)
    .nest("/api/auth", yauth_router.with_state(yauth_state.clone()))
    .layer(cors);
```

**`allow_credentials(true)` is mandatory** for the browser to send/receive cookies on cross-origin requests. Without it, `credentials: "include"` on the fetch side does nothing.

## 5. Cargo.toml Dependencies

Add yauth with the feature flags you need:

```toml
[dependencies]
yauth = { version = "0.3", features = ["email-password"] }
# Add more features as needed:
# yauth = { version = "0.3", features = ["email-password", "passkey", "mfa", "bearer"] }
```

You do **not** need the `openapi` feature flag in your app. That feature is only used internally by yauth's own CI to generate its `openapi.json` for the npm client package. Your app consumes the pre-built `@yackey-labs/yauth-client` npm package instead.

## 6. Database Migrations

yauth manages its own database tables (all prefixed with `yauth_`). Run yauth's migrations alongside your app's:

```rust
use yauth::migration::diesel_migrations;

// Run yauth migrations (with the features you enabled)
diesel_migrations::run_pending_migrations(&mut conn).await?;
```

If you want yauth's tables in a separate PostgreSQL schema (e.g., `auth`) to keep them isolated from your app tables:

```rust
let yauth_config = YAuthConfig {
    db_schema: "auth".into(),  // yauth tables go in the "auth" schema
    // ... other config
    ..Default::default()
};

// Use yauth::create_pool() which sets search_path automatically
let pool = yauth::create_pool(&database_url, &yauth_config)?;
```

## Summary

| Concern | Approach |
|---|---|
| **yauth router** | `nest("/api/auth", yauth_router.with_state(yauth_state))` |
| **Your app's OpenAPI spec** | Unchanged -- your utoipa + orval pipeline stays as-is |
| **yauth's TypeScript client** | Install `@yackey-labs/yauth-client` from npm, call `createYAuthClient({ baseUrl: "/api/auth" })` |
| **Session cookie on auth calls** | Handled automatically -- yauth's mutator defaults to `credentials: "include"` |
| **Session cookie on your app's calls** | Add `credentials: "include"` to your orval custom mutator |
| **CORS** | `allow_credentials(true)` + explicit origin list (no wildcards with credentials) |
| **Protecting your routes** | Apply `yauth::middleware::auth_middleware` via Axum layer, extract `Extension<AuthUser>` in handlers |
| **Database isolation** | Optional: set `db_schema: "auth"` in `YAuthConfig` to use a separate PG schema |

The two clients (your orval-generated one + yauth's pre-built one) coexist naturally. They share the same session cookie because they hit the same origin. Login via `authClient.emailPassword.login(...)` sets the cookie, and all subsequent requests from both clients carry it automatically.
